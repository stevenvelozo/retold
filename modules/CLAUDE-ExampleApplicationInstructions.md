# Building Example Applications for Retold Modules

How to add a runnable example to any Retold module so it works with `npx quack examples`. For the Pict view/template/CSS conventions used **inside** the example app, see [`modules/pict/CLAUDE.md`](pict/CLAUDE.md). This file only covers the harness around it.

## Pick the backend pattern first

Three patterns. Pick before you start; the structure is the same but the wiring differs.

| Pattern | When | Reference example |
|---|---|---|
| **Pure browser** | Demoing a Pict view, modal, animation тАФ anything that doesn't need server-side data or compute | [`pict-section-modal/example_applications/modal_garden`](pict/pict-section-modal/example_applications/modal_garden/) |
| **Built-in bookstore Meadow API** | Demoing CRUD-on-records (recordsets, forms, tables, joins). `quack examples-serve` provides a free SQLite-backed REST API at `/1.0/<Entity>` covering Book, Author, Customer, Review, etc. | [`pict-section-recordset/example_applications/bookstore`](pict/pict-section-recordset/example_applications/bookstore/) |
| **Bring your own Node API** | The example needs server-side compute that isn't CRUD (image processing, file conversion, ML inference, anything stateful that doesn't fit the bookstore schema) | [`retold-sharp/example_applications/sharp_playground`](utility/retold-sharp/example_applications/sharp_playground/) |

Most examples should use one of the first two. Reach for "bring your own" only when your demo genuinely needs custom server logic тАФ it costs you the ability to serve the example through the standard `quack examples` flow without `npm start`.

## Discovery rules (how `npx quack examples` finds examples)

- Looks at `<module>/example_applications/*/` subdirectories
- Each subdirectory must have a `package.json` with a `scripts.build` entry, otherwise it's skipped
- Build runs `npx quack build` (configured by quackage) per example
- Serve runs a single static HTTP server at the module level that exposes each example at `/<example-name>/index.html`

## Required structure

```
<module>/
тФЬтФАтФА package.json                                     # add "example": "npx quack examples" to scripts
тФФтФАтФА example_applications/
    тФФтФАтФА <example_name>/                              # snake_case
        тФЬтФАтФА package.json
        тФЬтФАтФА source/
        тФВ   тФЬтФАтФА Pict-Application-<Name>.js          # entry: registers views, exports default_configuration
        тФВ   тФФтФАтФА views/
        тФВ       тФФтФАтФА PictView-<Name>-Layout.js       # main layout view
        тФЬтФАтФА html/
        тФВ   тФФтФАтФА index.html                          # boots the bundled app
        тФФтФАтФА css/
            тФФтФАтФА <name>.css                          # styling (see "Header chrome" below)
```

Add `dist` and `node_modules` to the parent module's `.gitignore` if they aren't already (they're generated under `example_applications/*/`).

## Parent module wiring

In `<module>/package.json` scripts, add:

```json
"example": "npx quack examples"
```

That's it. No other changes needed at the module level.

## Example `package.json` (canonical)

```json
{
    "name": "<module>-example-<example-name>",
    "version": "0.0.1",
    "description": "<example display name> - short description",
    "main": "source/Pict-Application-<Name>.js",
    "scripts": {
        "start": "node source/server/<Name>-Server.js",
        "build": "npx quack build && npx quack copy",
        "test": "echo \"no tests\" && exit 0"
    },
    "author": "steven velozo <steven@velozo.com>",
    "license": "MIT",
    "dependencies": {
        "pict": "^1.0.348",
        "pict-application": "^1.0.28",
        "pict-view": "^1.0.67",
        "<parent-module>": "file:../../"
    },
    "devDependencies": {
        "quackage": "^1.0.59"
    },
    "copyFilesSettings": { "whenFileExists": "overwrite" },
    "copyFiles": [
        { "from": "./html/*", "to": "./dist/" },
        { "from": "./css/**", "to": "./dist/css/" },
        { "from": "./node_modules/pict/dist/*", "to": "./dist/js/" }
    ]
}
```

`start` is **optional** тАФ only needed for examples with a Node backend (see "Backend examples" below). Pure-browser examples don't need it.

## Naming conventions (these matter тАФ they wire into the boot HTML)

Given a package name `retold-sharp-example-sharp-playground`:

| Where | Form |
|---|---|
| Bundle file (browserify output) | `retold-sharp-example-sharp-playground.compatible.min.js` (kebab-case + `.compatible.min.js`) |
| Global JS variable (auto-generated by quack) | `RetoldSharpExampleSharpPlayground` (PascalCased) |

The HTML loads the file by kebab-case and boots it by PascalCase тАФ see template below.

Required pieces:

- `./js/pict.min.js` тАФ copied in by `quack copy`

- `<Name>-Application-Container` div тАФ the layout view's `ContentDestinationAddress`

## Pattern: Built-in bookstore Meadow API

`quack examples-serve` initializes a `retold-harness` instance with a SQLite-backed `bookstore` schema and exposes it as a REST API on the same port as the static example server. Your Pict views can talk to it directly with **zero backend code**.

### Entities exposed

All available at `/1.0/<EntityName>` and `/1.0/<EntityName>/:id`:

| Entity | Notes |
|---|---|
| `Book` | Title, ISBN, Genre, Language, plus the standard meadow housekeeping columns |
| `Author` | Joined to Book via `BookAuthorJoin` |
| `BookAuthorJoin` | Many-to-many bridge |
| `BookPrice` | Pricing history per book |
| `Review` | Per-book reviews |
| `BookStore`, `BookStoreInventory`, `BookStoreSale`, `BookStoreSaleItem`, `BookStoreEmployee` | Store/sales subdomain (extended seed data) |
| `Customer`, `User` | Tenant + identity (extended seed data) |

Standard meadow endpoints тАФ `GET /1.0/Book` returns a list, `GET /1.0/Book/:id` returns one, `POST` creates, `PUT` updates, `DELETE` deletes. Filter and pagination follow the meadow-endpoints conventions.

### Wiring the API into your views

In a `pict-section-recordset` view (or anywhere you'd point at a Meadow REST API):

```json
{
    "RecordSet": "Book",
    "RecordSetType": "MeadowEndpoint",
    "RecordSetMeadowEntity": "Book",
    "RecordSetURLPrefix": "/1.0/"
}
```

That's the entire backend hookup. The `bookstore` example wires several of these for `Book`, `Author`, and the `BookstoreInventory` virtual recordset тАФ copy [`Bookstore-Application.js`](pict/pict-section-recordset/example_applications/bookstore/Bookstore-Application.js) as the template.

### Storage

Quack writes the SQLite file to `<module>/example_applications/.data/bookstore.sqlite` and seeds it on first start. Add `.data/` to the parent module's `.gitignore`. Re-seeding is idempotent тАФ delete the file to reset.

### What this pattern is not for

If you need entities the bookstore doesn't have (your own domain model), or non-CRUD endpoints (image processing, ML inference, anything with custom request/response shapes), drop down to "bring your own Node API" below. Don't try to bend the bookstore schema to fit unrelated demos.

## Pattern: Bring your own Node API

For examples that need server-side compute that isn't CRUD on bookstore entities. Pattern:

1. Put the Node server at `source/server/<Name>-Server.js`
2. The server serves **both** the example's `dist/` (static) **and** its `/api/*` endpoints, on a single port
3. `package.json` тЖТ `scripts.start` runs that server
4. The view layer talks to `/api/*` via `fetch`

Copy [`Sharp-API-Server.js`](utility/retold-sharp/example_applications/sharp_playground/source/server/Sharp-API-Server.js) as the template. It handles static MIME types, request size limits, and the standard `sendJson`/`sendImage`/`sendError` helpers.

**Tradeoff to document in the example's README or in code comments:** `npx quack examples` only serves the static shell тАФ your `/api/*` endpoints don't exist there. For full functionality, users must `cd example_applications/<name> && npm start`. Worth saying out loud so people don't think the example is broken.

## Build & run flow

From the example directory:

```bash
npm install              # bring in pict, pict-application, the parent module via file:../../
npx quack build          # browserify тЖТ <name>.compatible.min.js (and friends)
npx quack copy           # html, css, pict.min.js тЖТ dist/
npm start                # (only if backend) launches the server
```

Or from the parent module:

```bash
npm run example          # тЖТ npx quack examples тЖТ discovers + builds + serves all examples
```

## Verification checklist

When you finish a new example, confirm:

- [ ] `dist/index.html` exists after `npx quack build && npx quack copy`
- [ ] `dist/js/pict.min.js` exists (proves the copy ran)
- [ ] `dist/<package-name>.compatible.min.js` exists (proves the bundle built)
- [ ] In the browser, the page loads with no console errors
- [ ] The status/header chrome renders (`.pict-example-header`)
- [ ] The layout view renders into `#<Name>-Application-Container`
- [ ] If there's an API: hitting an endpoint returns the expected payload

## Common mistakes

| Wrong | Right |
|---|---|
| Module's main `package.json` has no `example` script | Add `"example": "npx quack examples"` |
| Example's `package.json` is missing `build` | quackage skips it during discovery тАФ must have `scripts.build` |
| HTML loads `pict.min.js` from a CDN or relative parent path | Always `./js/pict.min.js` тАФ `quack copy` puts it there |
| Bundle filename in HTML doesn't match `package.json#name` | They must match; quack derives the bundle name from `name` |
| Boot global is wrong case | It's PascalCase of `name` (kebab тЖТ Pascal) |
| Backend code lives at module root, not in the example | Put it under `example_applications/<name>/source/server/` so the example is self-contained |
| Forgetting `node_modules/` and `dist/` in module `.gitignore` | Both are generated under `example_applications/*/` and should never commit |
| Building a custom Node API to serve a CRUD demo | The bookstore Meadow API at `/1.0/<Entity>` is free; use `RecordSetType: "MeadowEndpoint"` with `RecordSetURLPrefix: "/1.0/"` |
| Forgetting `.data/` in `.gitignore` (when using the bookstore API) | The SQLite file lands under `example_applications/.data/` and should never commit |
