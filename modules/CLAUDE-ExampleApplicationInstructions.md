# Building Example Applications for Retold Modules

How to add a runnable example to any Retold module so it works with `npx quack examples`. For the Pict view/template/CSS conventions used **inside** the example app, see [`modules/pict/CLAUDE.md`](pict/CLAUDE.md). This file only covers the harness around it.

## Pick the backend pattern first

Three patterns. Pick before you start; the structure is the same but the wiring differs.

| Pattern | When | Reference example |
|---|---|---|
| **Pure browser** | Demoing a Pict view, modal, animation — anything that doesn't need server-side data or compute | [`pict-section-modal/example_applications/modal_garden`](pict/pict-section-modal/example_applications/modal_garden/) |
| **Built-in bookstore Meadow API** | Demoing CRUD-on-records (recordsets, forms, tables, joins). `quack examples-serve` provides a free SQLite-backed REST API at `/1.0/<Entity>` covering Book, Author, Customer, Review, etc. | [`pict-section-recordset/example_applications/bookstore`](pict/pict-section-recordset/example_applications/bookstore/) |
| **Bring your own Node API** | The example needs server-side compute that isn't CRUD (image processing, file conversion, ML inference, anything stateful that doesn't fit the bookstore schema) | [`retold-sharp/example_applications/sharp_playground`](utility/retold-sharp/example_applications/sharp_playground/) |

Most examples should use one of the first two. Reach for "bring your own" only when your demo genuinely needs custom server logic — it costs you the ability to serve the example through the standard `quack examples` flow without `npm start`.

## Discovery rules (how `npx quack examples` finds examples)

- Looks at `<module>/example_applications/*/` subdirectories
- Each subdirectory must have a `package.json` with a `scripts.build` entry, otherwise it's skipped
- Build runs `npx quack build` (configured by quackage) per example
- Serve runs a single static HTTP server at the module level that exposes each example at `/<example-name>/index.html`

## Parent module wiring

In `<module>/package.json` scripts, add:

```json
"example": "npx quack examples"
```

That's it. No other changes needed at the module level.

## Example `package.json` (canonical)

```json
{
    "name": "<module>-example-<example-name>",
    "version": "0.0.1",
    "description": "<example display name> - short description",
    "main": "source/Pict-Application-<Name>.js",
    "scripts": {
        "start": "node source/server/<Name>-Server.js",
        "build": "npx quack build && npx quack copy",
        "test": "echo \"no tests\" && exit 0"
    },
    "author": "steven velozo <steven@velozo.com>",
    "license": "MIT",
    "dependencies": {
        "pict": "^1.0.348",
        "pict-application": "^1.0.28",
        "pict-view": "^1.0.67",
        "<parent-module>": "file:../../"
    },
    "devDependencies": {
        "quackage": "^1.0.59"
    },
    "copyFilesSettings": { "whenFileExists": "overwrite" },
    "copyFiles": [
        { "from": "./html/*", "to": "./dist/" },
        { "from": "./css/*", "to": "./dist/" },
        { "from": "./node_modules/pict/dist/*", "to": "./dist/" }
    ]
}
```

> **Copy pict + CSS to the dist *root*, not `./dist/js/` or `./dist/css/`.** Root paths serve fine under `quack examples` **and** let `prepare-docs` stage the example into your docs (see "Making an example stageable into docs" below). Subfolder paths silently break docs staging.

`start` is **optional** — only needed for examples with a Node backend (see "Backend examples" below). Pure-browser examples don't need it.

## Naming conventions (these matter — they wire into the boot HTML)

Given a package name `retold-sharp-example-sharp-playground`:

| Where | Form |
|---|---|
| Bundle file (browserify output) | `retold-sharp-example-sharp-playground.compatible.min.js` (kebab-case + `.compatible.min.js`) |
| Global JS variable (auto-generated by quack) | `RetoldSharpExampleSharpPlayground` (PascalCased) |

The HTML loads the file by kebab-case and boots it by PascalCase — see template below.

## Canonical `html/index.html`

```html
<!doctype html>
<html lang="en">
    <head>
        <meta charset="UTF-8" />
        <meta name="viewport" content="width=device-width, initial-scale=1.0" />
        <title><Display Name> - <module></title>
        <link rel="stylesheet" href="./<name>.css" />
        <style id="PICT-CSS"></style>
        <script src="./pict.min.js" type="text/javascript"></script>
        <script type="text/javascript">
//<![CDATA[
        Pict.safeOnDocumentReady(() => { Pict.safeLoadPictApplication(<PascalCaseName>, 2)});
//]]>
        </script>
    </head>
    <body>
        <div class="pict-example-header">
            <div class="pict-example-badge">
                <svg viewBox="0 0 16 16"><polygon points="8,1 10,6 16,6 11,9.5 13,15 8,11.5 3,15 5,9.5 0,6 6,6"/></svg>
                Pict Example
            </div>
            <div class="pict-example-app-name"><Display Name></div>
            <div class="pict-example-module"><module-name></div>
        </div>
        <div id="<Name>-Application-Container"></div>
        <script src="./<package-name-kebab>.compatible.min.js" type="text/javascript"></script>
    </body>
</html>
```

Required pieces:
- `<style id="PICT-CSS"></style>` — runtime CSS injection target
- `./pict.min.js` — copied in by `quack copy` (root-level path; required for docs staging — see below)
- `Pict.safeLoadPictApplication(<PascalCase>, 2)` — boots the app
- `<Name>-Application-Container` div — the layout view's `ContentDestinationAddress`

## Making an example stageable into docs

`npx quack examples` serves an example locally. `npx quack prepare-docs` (via pict-docuserve `stage-examples`) **additionally stages flagged examples into the module's `docs/` site** for GitHub Pages — they appear under `docs/examples/`, in the docs examples index, and on the landing-page splash. Two requirements:

**1. Flag it.** `stage-examples` only stages examples whose `package.json` carries the flag:

```json
"retold": {
    "ExampleApplication": {
        "Stage": true,
        "Title": "Modal Garden",
        "Summary": "One-line description shown in the docs examples index + splash.",
        "Complexity": "Basic | Intermediate | Advanced"
    }
}
```

`prepare-docs` then builds the example, copies its `dist/` into `docs/examples/<name>/`, and auto-maintains `docs/examples/README.md` (the index table) plus a marker-delimited region in `docs/_cover.md`. It also scaffolds a `docs/examples/<name>/README.md` writeup — only the generated launch block is rewritten, so hand-authored prose is preserved. Unflagged examples still run under `quack examples`; they just don't appear in the docs.

**2. Keep pict + CSS at the dist root** (the canonical `package.json` / `index.html` above already do — keep them that way). This is the trap: `stage-examples` does **not** recurse `dist/` subdirectories, and it only rewrites **root-level** pict references to the CDN. So an example authored with `./js/pict.min.js` + `./css/<name>.css` builds and serves fine under `quack examples` but **silently 404s when staged** — the `js/` / `css/` folders are never copied into `docs/`, and pict is never externalized. With root paths (`./pict.min.js`, `./<name>.css`), `prepare-docs` rewrites the pict tag to `https://cdn.jsdelivr.net/npm/pict@1/dist/pict.min.js` and stages your CSS + bundle next to `index.html`. Reference implementations: [`complex_table`](pict/pict-panel/example_applications/complex_table/) and [`shortcuts_playground`](pict/pict-provider-keybindings/example_applications/shortcuts_playground/).

After `prepare-docs`, confirm `docs/examples/<name>/index.html` loads pict from `cdn.jsdelivr.net` and the staged folder holds your `<name>.css` + `<bundle>.compatible.min.js` (and **no** `js/` / `css/` subfolders).

## Pattern: Built-in bookstore Meadow API

`quack examples-serve` initializes a `retold-harness` instance with a SQLite-backed `bookstore` schema and exposes it as a REST API on the same port as the static example server. Your Pict views can talk to it directly with **zero backend code**.

### Entities exposed

All available at `/1.0/<EntityName>` and `/1.0/<EntityName>/:id`:

| Entity | Notes |
|---|---|
| `Book` | Title, ISBN, Genre, Language, plus the standard meadow housekeeping columns |
| `Author` | Joined to Book via `BookAuthorJoin` |
| `BookAuthorJoin` | Many-to-many bridge |
| `BookPrice` | Pricing history per book |
| `Review` | Per-book reviews |
| `BookStore`, `BookStoreInventory`, `BookStoreSale`, `BookStoreSaleItem`, `BookStoreEmployee` | Store/sales subdomain (extended seed data) |
| `Customer`, `User` | Tenant + identity (extended seed data) |

Standard meadow endpoints — `GET /1.0/Book` returns a list, `GET /1.0/Book/:id` returns one, `POST` creates, `PUT` updates, `DELETE` deletes. Filter and pagination follow the meadow-endpoints conventions.

### Wiring the API into your views

In a `pict-section-recordset` view (or anywhere you'd point at a Meadow REST API):

```json
{
    "RecordSet": "Book",
    "RecordSetType": "MeadowEndpoint",
    "RecordSetMeadowEntity": "Book",
    "RecordSetURLPrefix": "/1.0/"
}
```

That's the entire backend hookup. The `bookstore` example wires several of these for `Book`, `Author`, and the `BookstoreInventory` virtual recordset — copy [`Bookstore-Application.js`](pict/pict-section-recordset/example_applications/bookstore/Bookstore-Application.js) as the template.

### Storage

Quack writes the SQLite file to `<module>/example_applications/.data/bookstore.sqlite` and seeds it on first start. Add `.data/` to the parent module's `.gitignore`. Re-seeding is idempotent — delete the file to reset.

### What this pattern is not for

If you need entities the bookstore doesn't have (your own domain model), or non-CRUD endpoints (image processing, ML inference, anything with custom request/response shapes), drop down to "bring your own Node API" below. Don't try to bend the bookstore schema to fit unrelated demos.

## Pattern: Bring your own Node API

For examples that need server-side compute that isn't CRUD on bookstore entities. Pattern:

1. Put the Node server at `source/server/<Name>-Server.js`
2. The server serves **both** the example's `dist/` (static) **and** its `/api/*` endpoints, on a single port
3. `package.json` → `scripts.start` runs that server
4. The view layer talks to `/api/*` via `fetch`

Copy [`Sharp-API-Server.js`](utility/retold-sharp/example_applications/sharp_playground/source/server/Sharp-API-Server.js) as the template. It handles static MIME types, request size limits, and the standard `sendJson`/`sendImage`/`sendError` helpers.

**Tradeoff to document in the example's README or in code comments:** `npx quack examples` only serves the static shell — your `/api/*` endpoints don't exist there. For full functionality, users must `cd example_applications/<name> && npm start`. Worth saying out loud so people don't think the example is broken.

## Build & run flow

From the example directory:

```bash
npm install              # bring in pict, pict-application, the parent module via file:../../
npx quack build          # browserify → <name>.compatible.min.js (and friends)
npx quack copy           # html, css, pict.min.js → dist/
npm start                # (only if backend) launches the server
```

Or from the parent module:

```bash
npm run example          # → npx quack examples → discovers + builds + serves all examples
```

## Verification checklist

When you finish a new example, confirm:

- [ ] `dist/index.html` exists after `npx quack build && npx quack copy`
- [ ] `dist/pict.min.js` exists (proves the copy ran)
- [ ] `dist/<package-name>.compatible.min.js` exists (proves the bundle built)
- [ ] In the browser, the page loads with no console errors
- [ ] The status/header chrome renders (`.pict-example-header`)
- [ ] The layout view renders into `#<Name>-Application-Container`
- [ ] If there's an API: hitting an endpoint returns the expected payload

## Common mistakes

| Wrong | Right |
|---|---|
| Module's main `package.json` has no `example` script | Add `"example": "npx quack examples"` |
| Example's `package.json` is missing `build` | quackage skips it during discovery — must have `scripts.build` |
| Hardcoding a CDN URL or parent path for `pict.min.js` in the HTML | Reference `./pict.min.js` (root) — `quack copy` provides it locally and docs staging rewrites it to the CDN |
| Loading pict or CSS from `./js/` or `./css/` subfolders | Keep pict + CSS at the dist **root** — `prepare-docs` doesn't stage `dist/` subdirs, so subfolder paths 404 on the published docs (see "Making an example stageable into docs") |
| Bundle filename in HTML doesn't match `package.json#name` | They must match; quack derives the bundle name from `name` |
| Boot global is wrong case | It's PascalCase of `name` (kebab → Pascal) |
| Backend code lives at module root, not in the example | Put it under `example_applications/<name>/source/server/` so the example is self-contained |
| Forgetting `node_modules/` and `dist/` in module `.gitignore` | Both are generated under `example_applications/*/` and should never commit |
| Building a custom Node API to serve a CRUD demo | The bookstore Meadow API at `/1.0/<Entity>` is free; use `RecordSetType: "MeadowEndpoint"` with `RecordSetURLPrefix: "/1.0/"` |
| Forgetting `.data/` in `.gitignore` (when using the bookstore API) | The SQLite file lands under `example_applications/.data/` and should never commit |
